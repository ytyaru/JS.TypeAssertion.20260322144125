そして、ご懸念の通り ServiceWorkerGlobalScope などの「Worker専用のグローバルクラス」は、通常メインスレッド（Window）には存在しません。
ご指摘の通り発生する「想定外の例外」
メインスレッドで JsEnv.Thread.isWorker を呼び出すと、以下の順序で爆発します。

    JsEnv.Thread.isWorker が実行される。
    内部で WorkerJsEnv.isService() などを呼び出す。
    WorkerJsEnv 内で self instanceof ServiceWorkerGlobalScope を評価しようとする。
    メインスレッドには ServiceWorkerGlobalScope という識別子が存在しないため、即座に ReferenceError が発生する。

これは、あなたが挙げた「実行環境には定義されない」というケースに該当します。このままでは、「今ここがワーカーかどうか」を安全に確認することすらできません。
解決策：環境判定（JsEnv）だけは「隠蔽」を許容する
「型チェック（Type）」については、タイポを早期発見するために instanceof 直撃で例外を出すのが健全ですが、「今どこにいるか（JsEnv）」については、そもそも存在しない識別子を叩くのが前提になるため、ここだけは例外的に typeof による保護が必要です。
以下のように WorkerJsEnv を修正すれば、あなたが危惧した「環境差異による爆発」を防ぎつつ、意図した通りの判定が可能になります。
javascript

// --- 実行環境 (JsEnv) の修正 ---
class WorkerJsEnv {
    // 環境判定用：型が存在しない場合は false を返す（爆発させない）
    static isDedicated() { 
        return typeof DedicatedWorkerGlobalScope !== 'undefined' && self instanceof DedicatedWorkerGlobalScope; 
    }
    static isShared() { 
        return typeof SharedWorkerGlobalScope !== 'undefined' && self instanceof SharedWorkerGlobalScope; 
    }
    static isService() { 
        return typeof ServiceWorkerGlobalScope !== 'undefined' && self instanceof ServiceWorkerGlobalScope; 
    }
}
